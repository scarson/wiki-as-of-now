# Online-eval batch — 2026-07-19 (batch 2)

**Protocol:** [2026-07-18-online-eval-protocol-design.md](../../design/2026-07-18-online-eval-protocol-design.md), including the 2026-07-19 stage-6 operational verdict vocabulary ([PR #64](https://github.com/scarson/wiki-as-of-now/pull/64)).
**Trigger:** the advisorySupport calibration change ([PR #63](https://github.com/scarson/wiki-as-of-now/pull/63) — hardened current-status definition + `Today:` data line in the triage prompt) landing on dev, per the protocol's change-triggered cadence. Sam approved the calibration design's four open questions 2026-07-19, including batch-2 timing ("as soon as the change lands") and directional measurement (no frozen-input replay).
**Primary question:** does the operationally-defined flag emit `false` at all (breaking batch 1's constant-true), and does the overclaim rate fall from the re-annotated 59% baseline — without selection loss (cards-per-pack collapse) or resolving-recall loss?

## Headline result

**The flag is still a constant. 24 of 24 cards carry `advisorySupport=true`; zero `false` emissions.** The hardened definition did not change the label's behavior, which is the design's own named failure criterion for option A and the stated trigger for **option D** (the categorical `resolving / related-only / unrelated` field).

The overclaim rate did fall sharply — **2 of 24 (8%) vs the re-annotated 59% baseline** — but **that drop is not attributable to the calibration change.** Retrieval got markedly better on this record set (22 of 24 proposed cards genuinely resolve their claim), so the model had only **two opportunities** to emit `false`, and it got both wrong. A flag that says `true` unconditionally scores well on a batch where almost everything really is resolving; that is a property of the sample, not of the model. Across both batches the model has now had **12 opportunities to emit `false` (10 in batch 1, 2 here) and has taken none.**

**Pipeline-under-test verification (done before drawing any conclusion, because "the deploy was stale" is the cheaper explanation and had to be ruled out):** the dev deploy run built commit `7f21bf3` (the PR #63 merge); that commit's `src/research/workers-ai-provider.ts` contains the hardened definition string; the "Deploy research worker" step completed **05:23:22Z** and a second deploy landed 05:29Z, both before the first batch-2 pack committed at **05:35:42Z**; and `workers/research/index.ts` imports the changed provider through `select-provider`. The packs ran on the new prompt.

## Findings

1. **Option A did not break the constant — the designated next step is now triggered.** Twelve chances across two batches, zero `false` emissions. The [calibration design](../../design/2026-07-18-advisory-support-calibration-design.md) names this exact outcome ("in particular, if the model still refuses to emit `false`") as the condition under which **option D — replacing the boolean with the categorical `resolving / related-only / unrelated`** — becomes the structural fix. The design's own reasoning for why D should work where A didn't remains intact: a category asks the model to *classify*, whereas a boolean it has already decided to propose invites self-consistent affirmation (the selection/labeling coupling). **Not implemented here:** option D changes the `cards_json` card shape inside write-once packs and ripples through `isProposalsShape`, `EvidenceCard`, `honesty-state.ts`, and the worksheet badge — a serialization-contract and data-integrity surface, which is a Review trigger. It is Sam's call, with this batch as the evidence.
2. **Option B′ is doing real work in the meantime, and its value is now measured rather than assumed.** Every one of the 24 cards would have lifted the claim-level worksheet state to "supported" under the old copy. Because the banner now reads "model suggests support — verify sources", the two overclaim cards present as an advisory suggestion rather than a verdict — and this applies at read time to every historical constant-era pack too.
3. **Source-quality gap recurs, second data point:** the K9 A1 record's only card is `handwiki.org` (a mirror of Wikipedia-derived content — arguably circular for a Wikipedia-sourcing tool), and one Gaganyaan card is `usthadian.com` (obscure aggregator) alongside four solid ones. This is the same shape as batch 1's `baike.baidu.com` finding and strengthens the case for the queued prefer-official retrieval-quality design, which now has three data points (baike, handwiki, usthadian).
4. **Retrieval breadth improved markedly and the messy stratum recovered** (2/2 vs batch 1's 0/2; zero `no_proposals` vs two). Attribution is uncertain — no pipeline change targeted retrieval between batches, so the most likely explanation is record difficulty, which is precisely why batch-over-batch rates are directional and why the *labeling* criterion (not the rate) was the pre-registered test.
5. **Definition edge case worth naming before batch 3:** the LRDR cards resolve the claim by reporting a 2020 cancellation — *earlier* than the claim's 2025 anchor year. The prompt's wording ("states the current status … as of a time after the claim's anchor year") reads as requiring post-anchor timing, yet a pre-anchor cancellation genuinely resolves a stale future-tense claim. Scored `resolving` here with the reasoning shown. If option D ships, its category definition should drop the temporal qualifier from the cancellation/supersession clause.
6. **Operational: the latency-episode failure mode recurred but was minor** (one job, ~7 minutes, zero cost) versus batch 1's two hour-scale episodes. Still no evidence justifying a resilience change; the probe pattern remains the cheap answer.
7. **Protocol: the confusion matrix earned its keep on its first use.** Reporting raw counts alongside the rate is what made the confound visible — a rate-only report would have recorded "59% → 8%, calibration works" and been wrong. The cards-per-pack guard also fired correctly (2.67 vs 1.7 = no selection loss), confirming the propose-anyway instruction did what it was written for even though the label did not move.
